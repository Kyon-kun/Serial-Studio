# Plan 0006 — Terminal scrollbar, configurable scrollback, and Text/Hex display labels

> **Phase 2 of 4 — the HOW.** The technical design that satisfies every requirement in
> [`spec.md`](./spec.md). Read the relevant `doc/claude/` sub-docs and the *actual code*
> before writing this — a plan grounded in a stale mental model is worse than no plan.
> Gate: do not start `/ss-tasks` until a human marks this `approved`.

## Approach (one paragraph)

Three independent slices, all main-thread UI. **Scrollbar:** extend the existing owner-drawn
`Widgets::Terminal::paintScrollbar()` into a track + thumb that renders whenever
`lineCount() > linesPerPage()` (autoscroll included), and add hit-testing to the widget's own
`mousePress/Move/ReleaseEvent` so a press on the scrollbar region enters drag/page mode
instead of text selection — mirroring the wheel handler's existing
disengage-on-scroll-up / re-engage-at-bottom autoscroll rules. **Scrollback:** replace the
`constexpr MAX_LINES = 1000` in `Terminal.cpp` with a `scrollbackLines` `Q_PROPERTY` on
`Console::Handler` (the settings owner: loaded/clamped in the ctor, persisted as
`Console/ScrollbackLines`), cached in the Terminal as `m_maxLines` and applied live via a
`scrollbackLinesChanged` connection that trims/reserves the buffers in lockstep; the
Settings dialog's Console tab gets a SpinBox following the existing Font Size pattern.
**Labels:** rename `displayModes()` to `tr("Text")` / `tr("Hex")` in `Handler.cpp`, and add
an exact-match `PINNED_TRANSLATIONS = {"Hex": "Hex"}` table to `llm_translate.py`, applied
during the pre-scan in `translate_ts_file()` so pinned strings are written directly and
never reach the LLM batch.

## Architecture & data flow

- **Scrollbar**: pure view-layer state inside `Widgets::Terminal` (QQuickPaintedItem, main
  thread). New members: `m_draggingScrollbar` (bool) and `m_dragAnchorY`/`m_dragAnchorOffset`
  (press-point → offset mapping so the thumb doesn't jump to the cursor). A
  `scrollbarHitRect()` helper computes the interactive band (thumb rect widened to ~12 px,
  full-height track strip on the scrollbar edge; left edge under RTL via the existing
  `m_translator.rtl()` convention in `paintScrollbar`). `mousePressEvent`: if the press falls
  in the band — thumb → start drag, track → page up/down by `linesPerPage()` — and *do not*
  start a selection; otherwise existing selection path runs unchanged. `mouseMoveEvent`
  routes to drag mapping while `m_draggingScrollbar`; `mouseReleaseEvent` ends it. Drag
  mapping inverts the paint mapping: `offset = round((thumbY / (availH - thumbH)) *
  (lineCount() - linesPerPage()))`, clamped. Any drag/page away from the bottom calls
  `setAutoscroll(false)`; reaching the max offset calls `setAutoscroll(true)` — the exact
  rules `wheelEvent` already implements. Repaints ride the existing `m_stateChanged` +
  `Misc::TimerEvents::uiTimeout` batching plus the immediate `update()` in
  `setScrollOffsetY`.
- **Scrollback**: `Console::Handler` owns the setting (same `QSettings` pattern as
  `Console/FontSize`). Each `Terminal` instance caches it in `m_maxLines` at construction
  and connects `scrollbackLinesChanged` → `applyScrollbackLimit()`: re-read, and if the
  buffer now exceeds the cap call `trimExcessLines(excess)` — the front-trim extracted from
  `appendString()` (text rows, color rows, cursor Y, scroll offset, **and selection points**,
  see Risks) — then `update()`. Raising the cap is bookkeeping-only; the buffer grows
  naturally on append. `initBuffer()` reserves `qMin(m_maxLines, 10000)` so a 100 k cap
  doesn't front-load a multi-MB reservation.
- **Labels/translation**: `displayModes()` strings flow unchanged through the existing
  QStringList → QML combo path (`NOTIFY languageChanged` retranslation already wired). After
  the rename, the maintainer's normal `llm_translate.py` run does lupdate (old entries
  pruned, new "Text"/"Hex" entries appear unfinished), then the pre-scan pins "Hex" in every
  locale before batching; "Text" translates normally. `en_US.ts` is already a direct copy.

## Hotpath & threading impact

- **Touches the hotpath?** No. `FrameReader` / `CircularBuffer` / `FrameBuilder` /
  `Dashboard` are untouched. The console display path is already decoupled from ingest:
  `Console::Handler::hotpathRxData` accumulates into `m_pendingDisplay`, flushed once per
  `uiTimeout` tick to `Terminal::append`. The only change on the append path is reading a
  cached `int` member (`m_maxLines`) instead of a `constexpr` — no allocation, no new
  signal, no per-append work that scales with buffer size. Lowering the cap does one
  proportional trim at the moment of change (spec constraint, satisfied). AC6 is covered by
  the CI `--benchmark-hotpath` gate as a no-regression check.
- **New cross-thread signal/slot?** No. `Console::Handler` and every `Terminal` instance
  live on the main thread; `scrollbackLinesChanged` is a same-thread auto connection.
- **New input to a cached hotpath flag?** No.
- **Timestamp ownership** — unaffected; no stamping anywhere in this change.

## Data model & persistence

- New `QSettings` key `Console/ScrollbackLines` (int, default 1000, clamped to
  [100, 100000] on load *and* in the setter, mirroring the FontSize clamp). No project-JSON,
  `Keys::`, schema, or Sessions DB impact. Older installs without the key get the default —
  identical to today's behavior.

## API / SDK surface

- None. `console.*` API handlers expose `displayMode` as an int plus the machine-facing
  `"PlainText"`/`"Hexadecimal"` identifiers — both unchanged. The scrollback setting is
  deliberately not exposed over the API (spec non-goal; can be added later as a
  `console.set` field if ever asked).

## QML / UI

- **Settings.qml Console tab, Display section**: one `Label` + `SpinBox` row
  (`from: 100`, `to: 100000`, `stepSize: 100`, `editable: true`), `value:
  Cpp_Console_Handler.scrollbackLines`, `onValueModified` writes back, `Connections` on
  `onScrollbackLinesChanged` restores — byte-for-byte the `_consoleFontSize` pattern, which
  also gives SpinBox-side clamping for AC2. Reset button gains
  `Cpp_Console_Handler.scrollbackLines = 1000`.
- **Terminal.qml**: no edits. Scrollbar drawing and input are C++-side; the right-button-only
  `MouseArea` and wheel/selection behavior are preserved.
- Scrollbar visuals reuse the existing palette (`QPalette::Window` thumb, same rounded
  style); the track is a faint strip in the same color at low alpha so it reads as a control
  without fighting the console theme. No hover animation (QQuickPaintedItem hover plumbing
  isn't worth it for a 6 px control; can be revisited).

## Tradeoffs & alternatives considered

| Decision | Options | Chosen + why |
|----------|---------|--------------|
| Scrollbar implementation | (a) owner-drawn in C++ paint + mouse handlers; (b) QML `ScrollBar` overlay bound to new notify properties; (c) Flickable rehost | **(a)** — single file, RTL comes from the existing convention, zero new per-tick property churn; (b) needs `lineCount`/`linesPerPage` notify plumbing firing every append batch plus manual RTL and z-order care over the selection MouseArea; (c) rewrites the render path. |
| Scroll unit | logical lines (status quo) vs. visual (wrap-aware) rows | **Logical lines** — `m_scrollOffsetY`, the wheel handler, and the paint mapping are all logical-line-based today; wrap-aware scrolling is a rendering-pipeline change the spec's non-goals exclude. The thumb mapping stays exactly as approximate as the current wheel scrolling. |
| Scrollback setting owner | (a) `Console::Handler` property; (b) `Terminal`-local property persisted by the widget | **(a)** — every console setting lives there, the Settings tab already binds `Cpp_Console_Handler`, and multiple `Terminal` instances stay in sync through one signal. |
| Pinning "Hex" | (a) deterministic `PINNED_TRANSLATIONS` exact-match table applied pre-batch; (b) `DOMAIN_GLOSSARY` prompt rule | **(a)** — the maintainer asked for a manual translation on exact match, and a prompt rule is probabilistic (the model may still translate or re-case it); the table is enforced again in `--verify-only` so it can't drift. |
| Buffer reservation at large caps | reserve full `m_maxLines` vs. cap the reserve | **Reserve `qMin(m_maxLines, 10000)`** — avoids a guaranteed multi-MB allocation for users who set 100 k "just in case"; QList growth beyond it is amortized and off the append fast path's hot loop. |

## Risks & mitigations

- **Selection indices after a trim (latent bug made hot).** `appendString()`'s existing trim
  adjusts cursor Y and scroll offset but *not* `m_selectionStart/End/StartCursor`; a stale
  selection Y past the shrunk buffer makes `copy()` index out of range. Today that needs
  1000+ lines to scroll past a live selection; a cap lowered from 100 k to 100 drops
  thousands of lines at once and makes it easy to hit. The extracted `trimExcessLines()`
  shifts selection points by the dropped count and clears the selection when it falls
  entirely off the front — fixing both the new path and the existing append path in one
  place. (Named here per the trust contract: it's a behavior fix inside code this change
  must extract anyway.)
- **Scrollbar drag vs. live appends.** While dragging, appends keep arriving and
  `lineCount()` grows, which would make a naive thumb mapping crawl under the cursor.
  Mitigation: drag disengages autoscroll immediately on thumb-press, and the drag mapping
  recomputes from the *current* `lineCount()` each move — the thumb tracks the user's
  pointer, not the moving content (standard terminal behavior).
- **Trim/color-row misalignment** — the shipped-bug class the spec names. Mitigation: one
  shared `trimExcessLines()` is the only trim site; it moves text rows, color rows, cursor,
  offset, and selection together (the existing `// code-verify off` lockstep comment moves
  with it). AC3 exercises it under streaming.
- **RTL hit-testing.** The scrollbar sits left under RTL; `scrollbarHitRect()` derives its X
  from the same `m_translator.rtl()` branch the painter uses, so paint and hit-test can't
  diverge. AC1 is verified in both directions.
- **SpinBox restore race** (`common-mistakes.md` setter-guard class): the handler setter
  guards on equality and clamps; the QML side uses `onValueModified` (user edits only) +
  `Connections` write-back — the same proven pattern as Font Size.
- **Translation regression on rename**: lupdate prunes the old "Plain Text"/"Hexadecimal"
  entries and the new pair ships unfinished until the maintainer's `llm_translate.py` run;
  the run itself pins "Hex" and LLM-translates "Text". Qt falls back to the English source
  for unfinished entries in the interim, so no locale ever shows an empty label (spec
  constraint).
